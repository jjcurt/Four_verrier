ℹ️ Format des programmes

Les programmes sont stockés au format JSON avec des étapes (steps) définissant les paramètres de cuisson.

Champs du programme :

  • name: Nom du programme
  • numSteps: Nombre d'étapes
  • manualMode (optionnel): true pour mode manuel (boucle ouverte, PID désactivé)
  • manualPWM (optionnel): PWM fixe 0-1023 en mode manuel
  • disablePidReset (optionnel, v1.6.1): true pour désactiver les resets PID entre étapes (meilleure continuité, risque de dépassement)

Logging de données - v1.6.0 :

  • enableDataLog (optionnel): true pour log utilisateur (macroscopique, 6 colonnes, défaut: 30s)
  • enableTestLog (optionnel): true pour log test PID (18 colonnes + diagnostics PID, défaut: 5s)
  • enableMaintenanceLog (optionnel): true pour log maintenance (21 colonnes + debug, défaut: 2s)
  • dataLogInterval (optionnel): Période d'échantillonnage en ms (défaut selon type de log)
  • adaptiveLogging (optionnel): true pour échantillonnage adaptatif intelligent (tous types)
  • adaptiveTempDelta (optionnel): Seuil de changement de température en °C (défaut: 1.0)
  • adaptiveDerivDelta (optionnel): Seuil de changement de dérivée en °C/min (défaut: 2.0)
  • adaptivePwmDelta (optionnel): Seuil de changement de PWM (défaut: 100)
  • adaptiveMinInterval (optionnel): Intervalle minimum entre échantillons en ms (défaut: 5000)

⚠️ IMPORTANT - Un seul type de log actif à la fois :

Pour garantir la robustesse du système, un seul fichier log peut être ouvert simultanément. Si plusieurs champs enableXxxLog sont à true, la priorité est : User > Test > Maintenance. Pour éviter toute confusion, activez un seul type de log par programme.

Types de logs :

  • User Log (user_*.csv) : Vue macroscopique pour les verriers (6 colonnes : Time, Temp, Target, SSR2_PWM, Phase, Reason)
  • Test Log (test_*.csv) : Tests de tuning PID (18 colonnes : + Kp, Ki, Kd, Error, Pterm, Iterm, Dterm, EffectiveHoldSec)
  • Maintenance Log (maint_*.csv) : Debug approfondi (21 colonnes : + FilteredTemp, StabilizingTimeSec, InitialBoost)

Champs de chaque étape :

  • targetTemp: Température cible (°C)
  • rampRate: Vitesse de montée (°C/min, 0 = palier uniquement)
  • withRamp: true si montée en température, false sinon
  • holdTime: Durée de maintien (minutes)

Exemple production (User Log) :

{"name":"Production_Verre","numSteps":3,"enableDataLog":true,"dataLogInterval":30000,"adaptiveLogging":true,"steps":[
  {"targetTemp":520,"rampRate":100,"withRamp":true,"holdTime":240},
  {"targetTemp":480,"rampRate":50,"withRamp":false,"holdTime":120},
  {"targetTemp":0,"rampRate":30,"withRamp":false,"holdTime":0}
]}

Exemple tuning PID (Test Log) :

{"name":"PID_Tuning_Test","numSteps":1,"enableTestLog":true,"dataLogInterval":5000,"steps":[
  {"targetTemp":200,"rampRate":100,"withRamp":true,"holdTime":60}
]}

Exemple identification boucle ouverte (Maintenance Log) :

{"name":"OpenLoop_PWM50_Debug","manualMode":true,"manualPWM":512,"enableMaintenanceLog":true,"dataLogInterval":2000,"adaptiveLogging":true,"numSteps":1,"steps":[
  {"targetTemp":0,"rampRate":0,"withRamp":false,"holdTime":60}
]}

💡 Note sur l'échantillonnage adaptatif :

Le mode adaptiveLogging réduit drastiquement le volume de données (~95-97%) tout en capturant les événements critiques : points d'inflexion, changements significatifs de température/PWM, zones de stabilisation. Idéal pour les tests expérimentaux longs et les programmes de production (12-48h) où seuls les événements importants doivent être archivés. Compatible avec les 3 types de logs.
